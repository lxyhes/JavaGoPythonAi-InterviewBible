系统设计面试题

涵盖高并发架构、分布式系统、微服务等设计类问题

设计原则

系统设计的步骤是什么？
重要
  1. 需求分析：功能需求、非功能需求（QPS、延迟、可用性）
  2. 估算：数据量、QPS、带宽、存储
  3. 高层设计：API设计、数据模型、基本架构
  4. 详细设计：组件详细设计、数据库设计、缓存策略
  5. 扩展性：如何处理高并发、故障恢复
如何设计一个高并发系统？
重要

分层优化：

  • 客户端：CDN、静态资源缓存、懒加载
  • 接入层：DNS轮询、负载均衡（LVS、Nginx）
  • 应用层：无状态设计、水平扩展、限流熔断
  • 服务层：微服务、服务降级、异步处理
  • 数据层：读写分离、分库分表、缓存

经典系统设计

如何设计一个短链接系统？
高频

核心问题：长URL映射到短URL

方案：

  • 发号器：自增ID，转换为62进制（a-zA-Z0-9）
  • 哈希：MD5后取前6位，冲突时+1重试

存储：

  • MySQL：短URL为唯一索引
  • Redis：缓存热点数据

跳转：301（永久，浏览器缓存）或302（临时，可统计）

如何设计一个秒杀系统？
高频

核心挑战：高并发、超卖、性能

设计方案：

  • 限流：验证码、答题、令牌桶限流
  • 削峰：消息队列（RocketMQ、Kafka）
  • 缓存：Redis预减库存，异步下单
  • 隔离：秒杀系统独立部署
  • 防超卖：Redis Lua脚本原子操作，数据库乐观锁
如何设计一个即时通讯系统？

架构：

  • 网关层：WebSocket长连接管理
  • 路由层：根据userId找到连接所在服务器
  • 业务层：消息处理、存储、推送

消息可靠性：

  • 消息确认机制（ACK）
  • 消息存储（离线消息）
  • 消息顺序（时间戳或自增ID）
如何设计一个分布式ID生成器？
高频

方案对比：

  • UUID：无序，不适合做数据库主键
  • 数据库自增：单点瓶颈
  • 雪花算法（Snowflake）：推荐，64位Long类型

雪花算法结构：

  • 1位符号位 + 41位时间戳 + 10位机器ID + 12位序列号
  • 支持69年，每秒每节点4096个ID

分布式系统

CAP理论是什么？
高频
  • C（Consistency）：一致性，所有节点数据一致
  • A（Availability）：可用性，每个请求都能得到响应
  • P（Partition Tolerance）：分区容错性，网络分区后仍能运行

定理：CAP无法同时满足，最多满足两个

选择：CP（ZooKeeper）、AP（Eureka）

BASE理论是什么？

BASE是CAP的延伸，强调最终一致性

  • BA（Basically Available）：基本可用
  • S（Soft State）：软状态，允许中间状态
  • E（Eventually Consistent）：最终一致性
一致性Hash是什么？解决什么问题？
高频

解决问题：普通Hash在节点增减时大量数据需要迁移

原理：

  • 将节点和数据都映射到一个Hash环上（0~2^32-1）
  • 数据顺时针找到第一个节点
  • 节点增减只影响相邻节点

虚拟节点：解决数据倾斜问题，一个物理节点对应多个虚拟节点

限流算法有哪些？
  • 计数器：固定窗口，简单但可能有突发流量
  • 滑动窗口：更平滑，但计算复杂
  • 令牌桶：以固定速率放入令牌，可应对突发流量
  • 漏桶：请求进入桶，以固定速率流出，平滑流量

消息队列

消息队列的作用？如何保证消息不丢失？
重要

作用：异步处理、应用解耦、流量削峰

保证不丢失：

  • 生产者：confirm机制，确保消息到达Exchange
  • MQ：消息持久化到磁盘
  • 消费者：手动ACK，处理完再确认
